## Tweet 2 -- Three-layer architecture

The Karpathy 3-layer architecture:

raw/ -- immutable JSONL-to-markdown conversion
wiki/ -- LLM-maintained entity/concept/source pages with [[wikilinks]]
site/ -- static HTML generated by one Python file

Each layer has one owner. No layer modifies an earlier one. Re-running any command is safe.

## Tweet 3 -- Adapter pattern

The adapter pattern for multi-agent support:

Each agent gets one ~50-line Python file that knows:
- Where the agent stores sessions
- How to discover .jsonl files
- How to derive a project slug

Everything else (parsing, filtering, redaction, rendering) is shared.

Adding a new agent = 1 file + 1 fixture + 1 test.

## Tweet 4 -- Pure-SVG visualizations

All charts are pure SVG generated at build time. No D3. No Chart.js.

viz_heatmap.py -- 365-day activity grid
viz_tools.py -- tool-calling bar charts
viz_tokens.py -- token usage stacked bars
changelog_timeline.py -- model changelog + pricing sparkline

CSS custom properties handle dark mode for all of them.

## Tweet 5 -- AI-consumable exports

Every page ships in three formats:

page.html -- for humans
page.txt -- plain text for LLMs
page.json -- structured metadata + body

Site-level: llms.txt, llms-full.txt, graph.jsonld, sitemap.xml, rss.xml

Plus an MCP server with 7 tools so Claude Desktop or Cursor can query your wiki live.

## Tweet 6 -- The stdlib-only constraint

One runtime dep: Python's `markdown` library.

HTML templates: f-strings
CSS: string constant in build.py
JS: string constant in build.py
SVG charts: stdlib Python generating strings
Frontmatter: re + str.split
Sitemaps: xml.etree
Hashing: hashlib
Dev server: http.server

No npm. No bundler. No Jinja. pip install -e . on Python 3.9+.
