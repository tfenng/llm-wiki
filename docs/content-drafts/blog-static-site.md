# Building a Static Site Generator in Python From Scratch

**Target:** Dev blog, Python community
**Length:** ~1,500 words
**Tone:** Technical, educational, opinionated

---

When I needed a static site generator for [llm-wiki](https://github.com/Pratiyush/llm-wiki), I built one from scratch in a single Python file. No Hugo. No Jekyll. No Eleventy. No Jinja templates. Just Python f-strings, the stdlib `markdown` library, and a deliberate set of constraints.

This post explains why, and walks through the implementation.

## Why not use an existing SSG?

The usual answer is "don't reinvent the wheel." But the wheel I needed had unusual specifications:

1. **Zero Node.js.** The project's entire user base is Python developers using AI coding assistants. Requiring `npm install` for the site generator would be a hard dependency on a toolchain many users do not have or want.

2. **Stdlib-only build.** The only runtime dependency should be Python's `markdown` library. No template engine. No asset pipeline. No YAML parser (frontmatter is parsed with `re` and `str.split`).

3. **Single-file generator.** The entire HTML rendering -- templates, CSS, JavaScript, SVG visualizations -- lives in one file (`build.py`). This means a contributor can read the whole rendering pipeline top to bottom without jumping between template files, config files, and plugin directories.

4. **Inline everything.** CSS is a string constant. JavaScript is a string constant. SVG charts are generated by Python functions that return strings. The output is self-contained HTML files with no external asset pipeline.

5. **Programmatic data sources.** The input is not a directory of markdown blog posts. It is structured session data with rich YAML frontmatter (project, model, tools_used, token_totals, tool_counts, duration_seconds) that needs to be aggregated, filtered, cross-referenced, and visualized. A template language would fight this; Python code embraces it.

Hugo, Jekyll, and Eleventy are excellent tools designed for blogs and documentation sites. llm-wiki is a **data application** that happens to output HTML. The rendering is the easy part; the data pipeline is where the complexity lives.

## The architecture

`build.py` does the following, in order:

1. **Discover** all `.md` files under `raw/sessions/` and `wiki/`
2. **Parse** frontmatter and body from each file
3. **Aggregate** per-project stats (session counts, tool totals, token sums, date ranges)
4. **Generate** computed data structures (heatmap grids, tool chart data, token timelines, model comparison pairs)
5. **Render** each page type using Python f-strings and the `markdown` library
6. **Write** the output to `site/`
7. **Generate** AI-consumable exports (llms.txt, JSON-LD, sitemap, RSS, per-page .txt/.json siblings)
8. **Build** the search index (JSON array of every page's title, project, date, and slug)

Steps 1-4 are pure data processing. Steps 5-8 are rendering. A template engine would only help with step 5, and would add complexity to everything else.

## CSS custom properties for theming

The CSS is organized around custom properties:

```css
:root {
  --bg: #ffffff;
  --text: #1e293b;
  --accent: #7c3aed;
  --bg-code: #f8fafc;
  --border: #e2e8f0;
  /* ... 40+ properties */
}

[data-theme="dark"] {
  --bg: #0f172a;
  --text: #e2e8f0;
  --accent: #a78bfa;
  --bg-code: #1e293b;
  --border: #334155;
}
```

Every color in the UI references a custom property. Dark mode is a single attribute toggle (`data-theme="dark"` on `<html>`) that reassigns every property. The JavaScript toggle writes to `localStorage` and respects `prefers-color-scheme` on first visit.

This approach extends to the SVG visualizations. The heatmap, tool charts, token cards, and pricing sparklines all use CSS custom properties for their colors:

```css
:root {
  --heatmap-0: #ebedf0;
  --heatmap-4: #216e39;
  --tool-cat-io: #3b82f6;
  --tool-cat-search: #8b5cf6;
  --token-input: #6366f1;
}
```

Because the SVG is inlined in the HTML (not an external `.svg` file), CSS custom properties work on SVG elements the same way they work on divs. One set of theme variables controls the entire page, including charts.

## Pure-SVG data visualization

This is the most unconventional part of the stack. Four Python modules generate SVG strings at build time:

**Activity heatmap** (`viz_heatmap.py`): A 365-day contribution grid. 53 columns of 7 cells each, Sunday-aligned. Quantile bucketing over non-zero days produces 5 intensity levels. The entire thing is a `<svg>` element with `<rect>` cells and `<title>` tooltips.

**Tool-calling bar chart** (`viz_tools.py`): Horizontal bars sorted by count, category-colored. Top 10 tools with an overflow row. Each bar is a `<rect>` with a `<text>` label and a tooltip showing count and percentage.

**Token usage card** (`viz_tokens.py`): Stacked bars for input, cache creation, cache read, and output tokens. A cache-hit-ratio badge. Human-readable K/M/B number formatting.

**Changelog timeline** (`changelog_timeline.py`): A vertical timeline of model entity changes with color-coded deltas and an inline pricing sparkline.

Why SVG instead of Chart.js or D3?

- **Zero JavaScript.** The charts render on page load with no script execution. They work in RSS readers, email clients, and `curl`.
- **Print-friendly.** SVG scales perfectly to any resolution. The print stylesheet works without special handling.
- **No external dependency.** The Python modules are stdlib-only. The output is a string that gets inlined into the HTML.
- **Dark mode for free.** CSS custom properties apply to SVG fills and strokes the same way they apply to background colors. No JavaScript theme callback needed.

The trade-off is interactivity. These charts do not zoom, pan, or animate. They are static snapshots of the data at build time. For a wiki of session transcripts, that is exactly the right level of interactivity.

## The JavaScript layer

The browser-side JavaScript is a single string constant embedded in `build.py`. It handles:

- **Theme toggle** with `data-theme` attribute, `localStorage` persistence, and `prefers-color-scheme` system preference
- **Command palette** (Cmd+K) with fuzzy matching over a pre-built search index, dataview-style structured queries, and wikilink hover previews
- **Keyboard shortcuts**: `/` (search), `g h` / `g p` / `g s` (navigation), `j` / `k` (table rows), `?` (help)
- **Filter bar** on the sessions table (project, model, date range, text)
- **Copy-as-markdown** and **copy-code** buttons using the Clipboard API with a `document.execCommand` fallback for HTTP
- **Auto-collapse** of long tool-result sections into `<details>` elements
- **Reading progress bar** linked to scroll position

Zero dependencies. No framework. No bundler. The entire file is about 400 lines of vanilla JavaScript.

## highlight.js for syntax highlighting

The one place where a client-side library won out over a build-time approach. Earlier versions used Pygments (Python) for server-side highlighting, but this added an optional dependency and made the build pipeline heavier.

The current approach loads highlight.js v11.9.0 from a pinned jsdelivr CDN. Two theme stylesheets (GitHub light and GitHub dark) are preloaded; the theme toggle swaps the `disabled` flag so code blocks stay in sync with the rest of the page. Code fences emit plain `<pre><code class="language-xxx">` and highlight.js handles the rest.

If the CDN is unreachable (offline use), code blocks render as plain monospace text. The degradation is graceful.

## What I would do differently

If starting over:

- **Separate the CSS into its own file.** Having it as a string constant in build.py makes diffs noisy. A real `.css` file with a build step to inline it would be cleaner.
- **Use a lightweight template function.** Not Jinja, but a 20-line `def render(template, **kwargs)` that handles escaping and variable substitution. Python f-strings work, but they make HTML templates hard to read.
- **Add incremental builds earlier.** Currently, `build.py` regenerates every page on every run. For small wikis (under 500 sessions) this is fast enough. For larger ones, incremental builds would cut iteration time.

These are refinements, not regrets. The core decision -- one file, stdlib-only, no template engine -- has held up well through 9 releases and 472 tests.

## Try it

```bash
git clone https://github.com/Pratiyush/llm-wiki.git
cd llm-wiki && ./setup.sh
./build.sh && ./serve.sh
```

The [live demo](https://pratiyush.github.io/llm-wiki/) is built from synthetic sessions on every push. The source for the entire site generator is one file: [`llmwiki/build.py`](https://github.com/Pratiyush/llm-wiki/blob/master/llmwiki/build.py).
